## 1. Project Overview

**Nacro Rewards** is a Microsoft Rewards automation suite composed of three independently-runnable products that share a common bot engine:

| Product | What it is |
|---|---|
| **Desktop App** | A native-feeling desktop window (Node.js + React) that controls a local Chromium/Patchright bot via a REST/IPC bridge |
| **Mobile App** | An Expo/React Native app that runs Bing searches and Daily Set tasks on Android/iOS using captured Microsoft cookies |
| **Bot Engine** | A TypeScript process (`src/`) that drives real Chromium browser sessions — used exclusively by the desktop app |

The three products share:
- **Account data** (`data/accounts.json`) — desk display store  
- **Session cookies** (`sessions/{email}/session_desktop.json`) — generated by either the bot engine or the cookie-capture flow  
- **Type definitions** from `lib/api-spec`, `lib/api-client-react`, and `lib/api-zod`

## 3. Bot Engine — `src/`

The bot engine is a long-running TypeScript process that controls real Chromium browsers via **Patchright** (a stealth fork of Playwright). It is launched either directly (`tsx src/index.ts`) or as a background subprocess spawned by the desk server.

---

### 3.1 Entry Point — `src/index.ts`

**Purpose:** Bootstraps the full bot lifecycle. Parses CLI flags, loads configuration and accounts, creates one `MicrosoftRewardsBot` instance per account, and hands off to the scheduler or runs immediately.

**Key responsibilities:**
- `--background` flag: runs as a background agent controlled via IPC (used by the desk server)
- Detects if running as a Node.js cluster worker or master
- Calls `runBackgroundAgent()` when `backgroundAgent.enabled = true` in config
- Account iteration order: optionally shuffled (`searchSettings.shuffleAccounts`)
- Applies per-account delay between runs (`searchSettings.accountDelay`)

**Exports:**
- `MicrosoftRewardsBot` — main orchestration class; ties all `core/` and `automation/` modules together for a single account

---

### 3.3 Account Schema — `src/accounts.json`

Each entry in the array represents one Microsoft account the bot will automate. `src/accounts.example.json` is the schema reference.

```jsonc
[
  {
    "email": "user@outlook.com",       // Required. Microsoft account email.
    "enabled": true,                   // false = skip this account on every run.
    "password": "secret",              // Required (unless using session cookies).
    "totpSecret": "BASE32SECRET",      // Optional. 2FA authenticator secret.
    "recoveryEmail": "backup@mail.com",// Optional. Fallback for verification challenges.
    "geoLocale": "en-US",             // Locale sent in browser/search requests.
    "langCode": "en",                  // Content language preference.
    "proxy": {                         // Optional per-account proxy.
      "url": "socks5://host",
      "port": "1080",
      "username": "u",
      "password": "p"
    },
    "saveFingerprint": {               // Cache and reuse browser fingerprint.
      "mobile": false,
      "desktop": false
    }
  }
]
```

> **Cookie-only accounts:** When an account was added via the "Capture Cookies" flow, `password` is an empty string. The bot's `ConfigLoader` detects a valid `sessions/{email}/session_desktop.json` and skips the login step entirely.

### 3.4 Core Layer — `src/core/`

#### `AgentRuntime.ts`
Runs a TCP IPC server when the bot is launched with `--background`. The desk server connects to it to send commands and receive log streams.

- `start()` — binds the IPC socket, writes address + auth token to `data/agent/agent.json`
- `stop()` — graceful shutdown
- **Messages handled:** `ping`, `run`, `stop`, `log-stream`
- **`data/agent/agent.json` format:** `{ port, token, pid, startedAt }`

#### `ActivityRunner.ts`
Dispatches individual reward tasks for a single account run. Reads the `workers` and `core` config sections to decide which tasks to execute. Distinguishes between:
- **Open-source tasks** — executed directly
- **Core (premium) tasks** — routed through `PluginManager` if a valid Core license is present

**Key method:** `runAll(account, browser)` — sequentially (or in parallel when safe) calls all enabled task handlers.

#### `SearchOrchestrator.ts`
Manages Bing search sessions. Fetches queries from `QueryProvider`, opens browser tabs, and paces searches according to `searchSettings`.

- Supports **parallel searching** (desktop + mobile simultaneously in separate contexts)
- Tracks how many searches remain until the daily cap is reached
- Reports progress to `LogService` after each individual search

#### `QueryProvider.ts`
Fetches search terms from multiple upstream sources, normalises them, and returns a shuffled pool.

**Sources (configured in `searchSettings.queryEngines`):**

| Key | Source |
|---|---|
| `google` | Google Trends daily trending searches |
| `wikipedia` | Wikipedia's "random" article title endpoint |
| `wikirandom` | Wikipedia random article opening sentence |
| `reddit` | Hot post titles from r/popular |
| `hackernews` | HN top story titles |
| `local` | Queries from `src/core/queries.json` / `search-queries.json` (bundled, offline) |

#### `Scheduler.ts`
Calculates when the next run should begin.

- Parses `scheduler.startTime` in the configured `timezone`
- Adds random jitter from `scheduler.randomDelay`
- Returns a `Date` object; the main loop sleeps until that time
- Handles DST transitions correctly via the IANA timezone database

#### `PluginManager.ts`
Sandbox system for loading extension modules (referred to as "Core" plugins).

- Validates plugin signatures before loading
- Anti-rollback: rejects plugin versions older than the last successfully loaded version
- Exposes `PluginAPI` to plugin code; plugins call back into the bot via `InternalPluginAPI`
- Plugins live in a sandboxed scope — they cannot access Node built-ins directly

#### `TaskBase.ts`
Abstract base class that all task implementations extend.

```typescript
abstract class TaskBase {
  abstract name: string;
  abstract run(page: Page, account: Account): Promise<void>;
  protected log(level, message): void;
  protected delay(min, max): Promise<void>;
}
```

Tasks in `src/core/tasks/`:
- `DailySet` — completes the 3-tile daily activity set
- `MorePromotions` — completes "More Promotions" tiles
- `PunchCards` — completes current punch card activities
- `DailyCheckIn` — clicks the daily check-in button
- `ReadToEarn` — opens and reads an article for points
- `ClaimPoints` — redeems accumulated point bonuses
- `ApplyCoupons` — applies available coupon codes

#### `SafetyAdvisory.ts`
Emits warnings when account-level settings look risky (e.g., no delays, too many clusters, proxy missing). Does not block execution.

#### `InternalPluginAPI.ts` / `PluginAPI.ts`
TypeScript interfaces that define the contract between the PluginManager sandbox and plugin code. `PluginAPI` is the public surface plugins can call; `InternalPluginAPI` is what the bot implements behind it.

### 3.5 Automation Layer — `src/automation/`

#### `BrowserManager.ts`
Central factory for all Chromium browser instances.

- Uses **Patchright** (stealth Playwright fork) to evade bot detection
- Per-account contexts: separate cookies, storage, UA string
- Proxy injection: reads account `proxy` config and passes to Playwright context options
- Calls `FingerprintManager` to inject a realistic fingerprint before any navigation
- Session restoration: loads `sessions/{email}/session_desktop.json` into the context on startup (skipping login if cookies are fresh)

#### `PageController.ts`
Low-level page interaction helper. All task code goes through this instead of calling Playwright directly.

**Key methods:**
- `navigate(url)` — goes to a URL, waits for network idle
- `click(selector)` — human-paced click with pre-click scroll into view
- `type(selector, text)` — types with randomised inter-key delays
- `waitForSelector(selector, timeout?)` — with fallback timeout
- `humanDelay(min, max)` — async sleep within the configured range

#### `FingerprintManager.ts`
Injects a realistic browser fingerprint into each new context using `fingerprint-generator` and `fingerprint-injector`.

- Fetches live User-Agent strings from Google/Microsoft UA hint APIs to stay current
- Generates fingerprint from a realistic browser + OS + screen resolution combination
- Caches fingerprints per account when `saveFingerprint.desktop = true`
- Mobile fingerprints use narrower screen + mobile UA

#### `BrowserViewport.ts`
Sets initial viewport and device pixel ratio to match the fingerprint.

#### `AutomationUtils.ts`
Static utility functions used across tasks:
- `randomBetween(min, max)` — integer in range
- `parseTimeString(str)` — converts `"30sec"`, `"1min"` etc. to milliseconds
- `waitRandom(page, min, max)` — sleeps a random duration within range
- `safeClick(page, selector)` — click with existence check + error suppression

#### `DashboardSelectors.ts`
Centralised CSS/XPath selector constants for the Microsoft Rewards dashboard. Maintains two sets:
- **Legacy** — classic ASP.NET dashboard
- **Next** — newer React-based dashboard

Auto-detected at runtime by checking for a discriminating element on the page.

#### `RewardsSidePanelController.ts`
Controls the slide-in rewards side panel that appears when clicking certain activity tiles.

#### `MagicCursor.ts`
Simulates natural mouse movement paths (Bezier curves) between elements to avoid straight-line cursor signatures that bot detectors flag.

#### `CorePromoBanner.ts`
Handles dismissal of interstitial promotional banners that sometimes appear before the dashboard loads.

#### `auth/` — Authentication Layer

| File | Purpose |
|---|---|
| `AuthManager.ts` | Top-level login orchestrator. Tries cookie login first; falls back to credential login if cookies are stale. Handles all challenge types. |
| `EmailAuth.ts` | Enters email + password on the Microsoft login form |
| `TotpAuth.ts` | Generates and enters TOTP codes using the account's `totpSecret` |
| `RecoveryAuth.ts` | Uses `recoveryEmail` to bypass SMS/app verification challenges |
| `SmsAuth.ts` | Handles SMS verification code prompt (manual or automated) |
| `AccountStateManager.ts` | Detects account lock, ban, or suspension states post-login |

Login flow:
1. Load `sessions/{email}/session_desktop.json` into browser context
2. Navigate to `baseURL` and check if already authenticated
3. If not: enter email → password → handle 2FA (TOTP / SMS / Recovery)
4. If banned/locked: log warning and skip account

#### `dashboard/`

| File | Purpose |
|---|---|
| `DashboardActions.ts` | Entry point — detects dashboard version and delegates |
| `LegacyDashboard.ts` | Task execution for the classic ASP.NET Rewards dashboard |
| `NextDashboard.ts` | Task execution for the newer React-based dashboard |

### 3.6 Helpers — `src/helpers/`

#### `ConfigLoader.ts`
Safe parser for `config.json` and `accounts.json`. Validates structure, applies defaults, and throws descriptive errors on malformed files.

- `loadConfig()` — returns the parsed `Config` object
- `loadAccounts()` — returns `Account[]` from `src/accounts.json`
- `loadSessionData(email, type)` — reads `sessions/{email}/session_{type}.json` (type = `"desktop"` | `"mobile"`)
- `saveSessionData(email, type, data)` — atomic write to the session file

#### `AtomicFile.ts`
Write-safe file operations: writes to a `.tmp` file then renames, preventing corrupt reads during concurrent writes.

- `write(filePath, content)` — atomic JSON write
- `read(filePath)` — safe read with parse error handling

#### `DataManager.ts`
Higher-level data access layer used by tasks. Abstracts `AtomicFile` for account stats, run logs, and session tracking.

#### `StatsRecorder.ts`
Records per-account run statistics:
- Points earned in this run
- Searches completed (desktop / mobile)
- Run duration
- Ban/lock detection flags

Writes to `data/run-logs.json` after each account finishes.

#### `HttpClient.ts`
Axios instance configured with:
- Proxy support (passes account proxy config when `proxy.queryEngine = true`)
- Retry logic for transient failures
- 30-second timeout

Used exclusively for non-browser HTTP requests (query fetching, webhook delivery).

#### `SchemaValidator.ts`
JSON Schema validation for config and account files. Produces human-readable error messages pinpointing the invalid field.

#### `ProxyUtils.ts`
Parses proxy URL strings (`socks5://user:pass@host:port`) into structured proxy config objects for Playwright.

#### `AvatarFetcher.ts`
Fetches the Microsoft account's profile picture URL from the Graph API (or from page scraping) and caches it alongside account data.

#### `PackageMetadata.ts`
Reads `package.json` for the current version string — used in User-Agent construction and log headers.

#### `HiddenDir.ts`
Creates and manages a hidden `.msrb/` directory inside the workspace for internal state that should not be git-tracked.

#### `Helpers.ts`
Miscellaneous shared utilities:
- `sleep(ms)` — promisified `setTimeout`
- `shuffleArray(arr)` — Fisher-Yates in place
- `chunkArray(arr, size)` — splits array into equal-size chunks
- `formatDuration(ms)` — converts milliseconds to a human-readable string (`"2m 34s"`)

#### `AccountSafetyWarning.ts`
Logs structured warnings when an account's configuration might cause detection or banning (e.g., no proxy on a flagged region, reused fingerprint across multiple accounts).

#### `AnalyticsService.ts` *(opt-in)*
When `analytics.enabled = true`, sends anonymised usage pings to the project's analytics endpoint. No account data or credentials are included. Can be disabled in `config.json`.

#### `ErrorReport.ts`
Formats exception stack traces into structured log objects for consistent error reporting across all modules.

### 3.7 Notifications — `src/notifications/`

#### `LogService.ts`
Central logging hub used by every other module.

**Log levels:** `debug`, `info`, `warn`, `error`

**Destinations:**
- **Console** (stdout) — filtered by `consoleLogFilter`
- **IPC stream** — pushed to connected desk server in real time
- **Webhook queue** — filtered by `webhookLogFilter`, forwarded to Discord/Ntfy
- **In-memory buffer** — last N lines kept for the desk's "Live Output" panel

**Key method:** `log(level, account, platform, title, message)`

#### `DiscordWebhook.ts`
Async queue for Discord webhook delivery. Batches rapid messages to stay under Discord's rate limit. Uses `HttpClient` for delivery.

- `send(payload)` — enqueues a Discord embed message
- Retries on 429 (rate limit) with the `retry_after` header value

#### `NtfyWebhook.ts`
Async queue for Ntfy push notification delivery. Uses standard HTTP PUT with Ntfy's JSON payload format.

- Supports `title`, `tags`, `priority`, `token` from config
- Queue prevents flooding when many accounts complete in succession

## 4. Desktop App — Desk Server

The desk server is a **zero-dependency Node.js HTTP server** at `scripts/desk/app-window.js`. It runs on port 3000 (configurable via `PORT` env var), serves the pre-built React UI from `dist-desk/`, and bridges the UI to the bot engine via TCP IPC.

Launch command:
```
node scripts/desk/app-window.js
```

---

### 4.1 `scripts/desk/app-window.js`

#### Constants

| Constant | Value | Description |
|---|---|---|
| `PORT` | `3000` (or `$PORT`) | HTTP listen port |
| `MSRB_TOKEN` | random 24-byte hex | Bearer token for API authentication (logged on start) |
| `OPEN_WINDOW` | `true` unless `$NO_WINDOW=1` | Whether to launch a Chrome/Edge app window |
| `WORKSPACE_ROOT` | `../../` relative to script | Absolute path to the monorepo root |
| `UI_DIST` | `{root}/dist-desk` | Served static files directory |
| `AGENT_STATE_FILE` | `data/agent/agent.json` | Where the bot writes its IPC address |
| `CAPTURE_DIR` | `data/agent/` | Directory for cookie-capture status files |

#### In-memory state

```javascript
const botState = {
```

#### IPC / Agent communication helpers

| Function | Description |
|---|---|
| `getAgentStatus()` | Reads `data/agent/agent.json`, opens a TCP socket, sends a `ping` message, returns `{ active, runState, ... }` |
| `sendRunCommand(targets)` | Sends a `run` message with the target account list over IPC |
| `sendStopCommand()` | Sends a `stop` message over IPC; returns `true` if the bot confirmed |
| `spawnBotProcess()` | Uses `spawn()` with the resolved `tsx` binary to launch `src/index.ts --background`; detached and unreffed so it outlives the desk server |
| `waitForAgent(maxMs)` | Polls `getAgentStatus()` every 500 ms until the agent responds or timeout |
| `resolveTsx()` | Finds the `tsx` binary: tries `node_modules/.bin/tsx.cmd` (Windows), `tsx` (Unix), then global `tsx` |
| `readCaptureStatus(file)` | Reads and parses a capture status JSON file; returns `null` on any error |

##### Bot Control

| Method | Path | Request Body | Response |
|---|---|---|---|
| `GET` | `/api/desk/status` | — | `BotStatus` (see §10) |
| `POST` | `/api/desk/run` | `{ accountIds?: string[] }` | `{ runId, started }` |
| `POST` | `/api/desk/stop` | — | `{ stopped: boolean, message }` |

**`POST /api/desk/run` logic:**
1. If the bot process is not already running, calls `spawnBotProcess()`
2. Waits up to 15 seconds for the agent IPC server to appear
3. Filters `accounts` to only `enabled: true` entries (or the specified `accountIds`)
4. Sends a `run` command via IPC
5. Updates `botState.isRunning = true` and returns a `runId`

**`GET /api/desk/status` logic:**
1. Pings the agent via IPC → `agentRunning`
2. Detects run completion: if `!agentRunning && botState.isRunning` → resets state, appends run log
3. Returns merged object of `botState` + live agent fields

##### Cookie Capture

| Method | Path | Request Body | Response |
|---|---|---|---|
| `POST` | `/api/desk/capture-session` | `{ email: string }` | `{ sessionId: string }` |
| `GET` | `/api/desk/capture-session/:id` | — | `CaptureStatus` |
| `DELETE` | `/api/desk/capture-session/:id` | — | `{ aborted: true }` |

**`POST` flow:**
1. Generates a random `sessionId` (6-byte hex)
2. Computes `statusFile = data/agent/capture-{sessionId}.json`
3. Spawns `tsx scripts/desk/cookie-capture.ts {sessionId} {email} {statusFile}`
4. Registers `child`, `statusFile`, `email` in `captureSessions` Map
5. On subprocess exit: if status is not `"done"`, overwrites status file with `{ status: "failed" }`
6. On subprocess error: writes error to status file immediately

**`DELETE` flow:**
1. Kills the subprocess if still running
2. Removes from `captureSessions` Map
3. Deletes the status file from disk

### 4.3 `scripts/desk/cookie-capture.ts`

Standalone Patchright script spawned as a subprocess by `app-window.js`. Not imported — always run via `tsx`.

**Arguments:** `sessionId`, `email`, `statusFilePath`

#### Flow

```
1. Write status: { status: "opening" }
2. Launch a VISIBLE Chromium browser (headless: false)
3. Navigate to: https://login.live.com/oauth20_authorize.srf?...
   (Microsoft login page — Rewards scope)
4. Write status: { status: "waiting" }
5. Wait up to 10 minutes for the URL to land on:
   - rewards.microsoft.com (not /auth, not /login)
   - OR account.microsoft.com (not /login)
6. Write status: { status: "capturing" }
7. Call context.cookies() — captures ALL cookies across all domains
8. Create sessions/{email}/ directory if needed
9. Write cookies to: sessions/{email}/session_desktop.json
10. Write status: { status: "done", cookieCount: N, sessionFile: "..." }
11. Close browser, exit 0
```

**On any error:** writes `{ status: "failed", error: message }` to the status file, exits 1.

#### Status file format (`data/agent/capture-{sessionId}.json`)

```jsonc
{
  "sessionId": "a1b2c3",
  "email": "user@outlook.com",
  "status": "waiting",           // opening | waiting | capturing | done | failed
  "cookieCount": 42,             // present only when status = "done"
  "sessionFile": "sessions/...", // present only when status = "done"
  "error": "timed out"           // present only when status = "failed"
}
```

#### `useAccounts()`

```typescript

  isLoading,     // boolean
  error,         // Error | null
  addAccount,    // UseMutationResult — POST /api/desk/accounts
  deleteAccount, // UseMutationResult — DELETE /api/desk/accounts/:id

```

#### `useBotStatus()`

```typescript
const {
  status,    // BotStatus | undefined
  isLoading, // boolean
  runNow,    // UseMutationResult — POST /api/desk/run

} = useBotStatus();
```

- Polls `GET /api/desk/status` every 2 seconds
- `runNow.mutate({ accountIds? })` — triggers a run, invalidates status + accounts

#### `useCookieCapture()`

```typescript
const {
  start,         // UseMutationResult — mutate(email: string)
  abort,         // UseMutationResult — mutate() — kills subprocess
  reset,         // () => void — clears session state without server call

  captureStatus, // CaptureStatus | null
```

Shows next scheduled run time and a countdown.

### 6.2 State & Context — `context/`

#### `AccountsContext.tsx`

Provides the full account and run-log state to the app.

**State shape:**
```typescript
interface Account {
  id:             string;
  email:          string;
  name:           string;
  avatar?:        string;          // Profile picture URL
  cookies:        string;          // Raw cookie string for fetch headers
  cookiesSyncedAt?: string;        // ISO timestamp
  enabled:        boolean;
  status:         'idle' | 'running' | 'done' | 'failed' | 'banned';
  todayPoints:    number;
  totalPoints:    number;
  searchesDone:   number;
  lastRun?:       string;
}

interface RunLog {
  id:          string;
  accountId:   string;
  accountName: string;
  startedAt:   string;
  finishedAt:  string;
  pointsEarned: number;
  searchesDone: number;
  status:      'success' | 'partial' | 'failed';
  error?:      string;
}
```

**Methods exposed via context:**
- `addAccount(account)` — appends and persists
- `updateAccount(id, patch)` — merges patch and persists
- `removeAccount(id)` — removes and persists
- `setAccountStatus(id, status)` — lightweight status update
- `recordPoints(id, points)` — increments today/total points
- `appendRunLog(log)` — adds to run log, trims to 200 entries
- `clearRunLogs()` — empties the log
- `syncCookies(id, cookieString)` — updates cookie string + `cookiesSyncedAt`

**Persistence:** All state is serialised to AsyncStorage under `@nacro/accounts` and `@nacro/run-logs`.

#### `LicenseContext.tsx`

Manages the license key system that controls feature access.

**Tiers:**

| Tier | Account Limit | Features unlocked |
|---|---|---|
| `basic` | 3 accounts | Searches + Daily Set |
| `core` | 20 accounts | All features + custom queries + background mode |

**Methods:**
- `activateLicense(key)` — validates key against remote API; sets `tier` and `accountLimit`; persists to AsyncStorage
- `isFeatureEnabled(feature)` — returns boolean for a named feature gate

**Validation:** key is sent to the license API endpoint; response contains `{ valid, tier, accountLimit }`.

#### `SettingsContext.tsx`

Global automation parameters, persisted to AsyncStorage.

| Setting | Type | Default | Description |
|---|---|---|---|
| `desktopSearchCount` | number | 30 | PC searches per account per run |
| `mobileSearchCount` | number | 20 | Mobile searches per account per run |
| `searchDelayMin` | number | 2 | Seconds to wait between searches (min) |
| `searchDelayMax` | number | 5 | Seconds to wait between searches (max) |
| `parallelSearch` | boolean | true | Run desktop + mobile simultaneously |
| `overnightEnabled` | boolean | false | Enable background overnight mode |
| `overnightStartTime` | string | `"02:00"` | Overnight window start (HH:MM) |

**Methods:** `updateSetting(key, value)` — sets and persists a single setting.

#### `QueriesContext.tsx`

Manages the rotating pool of Bing search queries.

**State:**
- `unusedQueries: string[]` — queries not yet searched today
- `usedQueries: string[]` — queries already submitted

**Methods:**
- `popQuery()` — removes and returns next unused query; moves it to `usedQueries`
- `addQuery(text)` — adds a new query to `unusedQueries` (Core only)
- `resetQueries()` — moves all `usedQueries` back to `unusedQueries`
- `shuffleQueries()` — Fisher-Yates shuffle of `unusedQueries`

Default query pool: bundled list of ~200 general-interest terms, loaded on first install.

### 6.3 Utilities — `utils/`

#### `utils/bingSearch.ts`

Executes a single Bing search as a plain HTTP `fetch()` request — no browser required.

```typescript
async function bingSearch(options: {
  query:   string;
  cookies: string;        // Cookie header value from AccountsContext
  mobile?: boolean;       // true = mobile User-Agent
  cvid?:   string;        // Bing client visit ID (randomised per search)
}) → Promise<{ success: boolean; points?: number; error?: string }>
```

**How it works:**
1. Constructs `https://www.bing.com/search?q={query}&cvid={random}` URL
2. Sets realistic headers: `Cookie`, `User-Agent`, `Accept-Language`, `Referer`
3. Parses the response to detect reward points delta (looks for `reward` JSON island in HTML)
4. Returns success/failure + points earned

#### `utils/backgroundSearch.ts`

Registers and manages background search tasks for overnight mode using `expo-task-manager` and `expo-background-fetch`.

**Exported task name:** `OVERNIGHT_SEARCH_TASK`

**Task behaviour:**
1. Wakes at the scheduled overnight start time
2. Checks `SettingsContext.overnightEnabled`
3. Iterates enabled accounts from `AccountsContext`
4. Calls `bingSearch()` for each remaining unused query
5. Updates account stats and appends run log
6. Reschedules the task for the next slot (if duration window is active)

**Functions:**
- `registerOvernightTask(startTime, durationHours)` — registers the background fetch task
- `unregisterOvernightTask()` — cancels the registered task
- `getNextRunTime()` — returns the `Date` of the next scheduled overnight window

#### `utils/notifications.ts`

Manages local push notifications for background run feedback.

- `showProgressNotification(accountName, progress)` — persistent "sticky" notification with progress percentage; updated in place via notification ID
- `showCompletionNotification(summary)` — one-shot notification on run completion with point totals
- `cancelProgressNotification()` — dismisses the sticky progress notification
- `requestPermissions()` — prompts the user for notification permission

## 7. Shared Libraries — `lib/`

### `lib/api-spec/`

Contains the OpenAPI 3.1 specification (`openapi.yaml`) — the single source of truth for all desk API types.

**Code generation:** `orval.config.ts` drives `orval` to generate:
- `lib/api-client-react/` — TanStack Query hooks + Axios client
- `lib/api-zod/` — Zod schemas for runtime validation

**Defined schemas:**

| Schema | Description |
|---|---|
| `DeskAccount` | Full account record returned by the API |
| `DeskAccountInput` | Fields accepted when creating an account |
| `BotStatus` | Current run state and stats |
| `RunInput` | Body for `POST /desk/run-now` |
| `RunResult` | Response from a run trigger |
| `RunLog` | Single historical run record |
| `DeleteResult` | `{ deleted: boolean }` |
| `HealthStatus` | `{ ok: boolean }` |

### `lib/api-client-react/`

Generated TanStack Query hooks + plain Axios functions. Used exclusively by the desk UI.

**Generated hooks (relevant to desk UI):**
- `useListAccounts()` — query hook for `GET /desk/accounts`
- `useDeleteAccount()` — mutation hook for `DELETE /desk/accounts/:id`
- `useRunNow()` — mutation hook for `POST /desk/run-now`
- `useGetBotStatus()` — query hook for `GET /desk/status`
- `useGetRunLogs()` — query hook for `GET /desk/logs`

### `lib/api-zod/`

Generated Zod validators matching every OpenAPI schema. Used server-side for request validation and client-side for runtime type checking.

### `lib/db/`

Drizzle ORM schema and migrations for a PostgreSQL database. Not used by the current desk server (which uses flat JSON files), but present for a future API server artifact.

## 8. Data Structures & File Formats

### `data/accounts.json` — Desk Account Store

```jsonc
[
  {
    "id":               "uuid-v4",
    "email":            "user@outlook.com",
    "name":             "Primary Account",
    "status":           "idle",          // idle | running | done | failed | banned | locked
    "todayPoints":      0,
    "totalPoints":      0,
    "searchesCompleted":0,
    "lastRun":          null,            // ISO 8601 or null
    "enabled":          true,
    "createdAt":        "2026-07-18T..."
  }
]
```

### `data/run-logs.json` — Run History

```jsonc
[
  {
    "id":           "run-a1b2c3d4",
    "startedAt":    "2026-07-18T08:00:00Z",
    "finishedAt":   "2026-07-18T09:12:00Z",
    "accountsRun":  3,
    "status":       "success",          // success | partial | failed
    "accountResults": [
      {
        "accountId":    "uuid-v4",
        "name":         "Primary Account",
        "pointsEarned": 85,
        "searchesDone": 50,
        "status":       "done"
      }
    ]
  }
]
```

### `data/agent/agent.json` — Bot IPC Descriptor

Written by `AgentRuntime.start()`. Read by the desk server to connect via TCP.

```jsonc
{
  "port":      49152,
  "token":     "hex-auth-token",
  "pid":       12345,
  "startedAt": "2026-07-18T08:00:00Z",
  "runState":  "idle"              // idle | running | stopping
}
```

### `data/agent/capture-{sessionId}.json` — Cookie Capture Status

Transient file created by `cookie-capture.ts`. Deleted when the desk server handles `DELETE /api/desk/capture-session/:id`.

```jsonc
{
  "sessionId":   "a1b2c3",
  "email":       "user@outlook.com",
  "status":      "done",           // opening | waiting | capturing | done | failed
  "cookieCount": 42,
  "sessionFile": "sessions/user@outlook.com/session_desktop.json",
  "error":       null
}
```

### `sessions/{email}/session_desktop.json` — Desktop Session Cookies

Array of Playwright/Patchright cookie objects. Format is the standard Playwright `Cookie[]` shape.

```jsonc
[
  {
    "name":     "MUID",
    "value":    "...",
    "domain":   ".bing.com",
    "path":     "/",
    "expires":  1753000000,
    "httpOnly": false,
    "secure":   true,
    "sameSite": "None"
  }
]
```

### `sessions/{email}/session_mobile.json`

Same format as `session_desktop.json`. Used by the bot engine when running mobile searches. Generated by the mobile app's `login-webview.tsx` if the cookie sync feature is used.

```
# 1. Install dependencies
pnpm install
```

## 10. API Reference

All endpoints are served by `scripts/desk/app-window.js` on port 3000. The `Authorization: Bearer {token}` header is required on every `/api/` request. The token is printed to the console on startup and returned by `GET /api/token`.
